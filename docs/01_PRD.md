# Edgechat 产品需求文档（PRD）

版本：v2.0.0
状态：生效中
最后更新：2026-05-03

## 1. 背景与业务目标

### 1.1 背景
传统自建 IM 系统需要大量服务器成本和运维投入。Cloudflare 提供的边缘计算能力和 Durable Objects 可以支撑低成本的实时通信系统。

### 1.2 业务目标
在 Cloudflare 生态内建立一套可直接部署的、低运维成本的站内即时通讯（IM）系统，支持企业或社区的内部沟通需求。

### 1.3 目标用户
- 小型企业和初创团队
- 开源社区
- 内部协作工具需求者

## 2. 功能清单（In Scope）

### 2.1 账号与身份管理
- 仅管理员创建用户账号（不开放自助注册）
- 用户登录（用户名 + 密码）
- 用户信息管理（头像、昵称、个性签名）
- 密码重置（仅管理员执行）
- 会话管理（HttpOnly Cookie，自动过期）
- 角色定义：普通用户、管理员

### 2.2 消息与沟通
- 公开群组：所有成员可见，所有人可加入
- 私有群组：仅成员可见，需群主邀请
- 私信（DM）：一对一实时对话
- 实时消息推送（基于 WebSocket）
- 历史消息分页查询
- 消息内容支持：纯文本、Markdown、文件
- 消息搜索（按关键词）

### 2.3 群组管理
- 群主创建群组
- 群主管理成员（添加、移除）
- 群主修改群组名称、描述、头像
- 管理员可查看任意群组（不限权限）
- 管理员可查看任意私信（审计用途）
- 群组软删除后自动硬删除（可配置保留天数）

### 2.4 文件与多媒体
- 文件上传（限制文件类型与大小）
- 文件下载与在线预览
- 头像管理（个人 + 群组）
- 危险文件扩展名黑名单（.php, .exe, .sh 等）

### 2.5 后台管理
- 用户管理页面：查看、编辑、删除用户
- 消息管理页面：查看、搜索、删除消息
- 网站设置页面：全局配置
- 审计日志：所有管理操作记录

### 2.6 系统功能
- 定时清理过期消息（软删除后）
- 定时清理失效会话
- 消息速率限制（防止刷屏）
- 文件大小限制（20MB）

## 3. 明确不做（Out of Scope）

- 自助用户注册
- 第三方 OAuth（仅支持用户名密码）
- 端到端加密
- 消息转发与多语言支持
- 视频 / 音频通话
- 公开 API（仅内部使用）

## 4. 用户故事（Acceptance Criteria）

### 4.1 用户登录
作为一个员工，我想用用户名和密码登录系统，以便访问团队沟通。

验收标准：
- 正确的用户名和密码可成功登录
- 错误凭证返回明确的错误消息
- 登录后自动创建 7 天有效期的会话
- 会话信息存储在 HttpOnly Cookie 中
- 服务端需验证会话未过期

### 4.2 发送消息
作为群组成员，我想在群组或私信中发送消息，以便与他人沟通。

验收标准：
- 消息被实时推送到所有在线成员（延迟 < 100ms）
- 离线成员在上线时可加载历史消息
- 消息内容被清理（防止 XSS 攻击）
- 消息大小限制为 10KB
- 用户每秒最多发 10 条消息（速率限制）
- 超限消息被拒绝并返回错误

### 4.3 创建群组
作为用户，我想创建私有群组，以便与特定人员合作。

验收标准：
- 创建群组成功返回群组 ID
- 创建者自动成为群主
- 私有群组仅成员可见
- 群主可邀请成员（不接受自动加入请求）

### 4.4 管理员删除用户
作为管理员，我想删除用户账号，以便管理团队成员。

验收标准：
- 删除操作记录在审计日志中（含操作者、时间、IP）
- 不允许删除自己
- 不允许删除超级管理员
- 删除用户后其所有消息被软删除
- 操作返回明确的成功或错误提示

## 5. 非功能要求

### 5.1 性能
- 消息端到端延迟：< 100ms（在相同地域）
- 页面加载时间：< 2s
- API 响应时间：< 500ms（99 百分位）
- 支持并发用户数：最少 1000+

### 5.2 可用性
- 系统可用性：99% 以上
- 自动故障转移（Cloudflare 处理）
- 消息持久化，不丢失关键数据

### 5.3 安全与合规
- HTTPS/TLS 加密传输
- HttpOnly 与 SameSite Cookie 强制
- CORS 白名单控制
- 所有管理操作需审计日志
- 敏感数据（密钥、密码）不存储明文
- 用户密码需 PBKDF2 或 Argon2 加密

### 5.4 扩展性
- 数据库设计支持单表 1000 万+ 行
- 消息检索需索引优化
- 支持水平扩展（无状态设计）

## 6. 验收条件

全部功能必须满足：
- 功能完整性：所有 2.1-2.6 功能可用
- 安全审计：0 个高危及以上漏洞
- 测试覆盖：核心路径 > 80% 代码覆盖率
- 性能基准：满足 5.1 性能目标
- 文档完整：API 文档、部署指南、运维手册齐全
