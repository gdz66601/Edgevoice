## 1. 监测与日志

### 1.1 关键指标（KPI）

需要持续监测的指标：

**可用性指标**：
- 系统可用性：目标 > 99%
- 平均故障恢复时间（MTTR）：目标 < 1 小时
- 平均故障间隔（MTBF）：目标 > 30 天

**性能指标**：
- API 响应时间：p50 < 200ms，p99 < 500ms
- WebSocket 消息延迟：< 100ms
- 页面加载时间：< 2 秒
- 数据库查询时间：p99 < 100ms

**错误率指标**：
- 全局错误率：目标 < 0.1%
- 5xx 错误率：目标 < 0.01%
- 登录失败率：< 1%
- 文件上传失败率：< 0.5%

**业务指标**：
- 日活用户数（DAU）
- 消息吞吐量（消息/秒）
- 存储使用量
- 网络流量

### 1.2 日志收集

**日志来源**：
- Cloudflare Workers 日志
- D1 数据库查询日志
- 应用审计日志（admin_audit_log 表）

**日志级别**：
- ERROR：异常、业务错误
- WARN：潜在问题、降级
- INFO：重要事件（登录、删除）
- DEBUG：详细信息（用于故障排查）

**日志查看**：
```bash
# 实时日志
wrangler tail --env production

# 查询历史日志（通过 Cloudflare 仪表板）
# Logs -> Workers
```

### 1.3 日志分析

常见的需要关注的日志模式：

**异常登录**：
```

```

**数据库异常**：
```

```

**权限问题**：
```

```

**速率限制**：
```
WARN: 用户 789 消息速率限制，尝试发送 50 消息/秒
```

## 2. 告警与通知

### 2.1 告警规则

**P0 告警（立即响应）**：
- 系统不可用（连续 5 分钟 5xx 错误 > 50%）
- 数据库连接失败
- R2 存储不可用
- WebSocket 断开率 > 50%

**P1 告警（1 小时内响应）**：
- 错误率 > 1%
- API 响应时间 p99 > 1s
- 登录失败率 > 5%
- 存储使用量 > 80%

**P2 告警（4 小时内响应）**：
- 错误率 > 0.1%
- API 响应时间 p50 > 500ms
- 离线消息队列堆积 > 10000
- CPU 时间使用 > 70%

### 2.2 通知渠道

- P0：短信 + 电话 + Slack
- P1：电邮 + Slack
- P2：Slack 频道

### 2.3 告警示例

使用 Cloudflare Workers Analytics 或第三方服务（如 Sentry、DataDog）配置告警。

## 4. 故障处理

### 4.1 故障分类

**网络故障**：
- 工作域 DNS 无法解析
- TLS 证书过期
- CDN 无法访问

**应用故障**：
- Worker 脚本错误
- 内存不足
- 无限循环

**数据库故障**：
- D1 连接失败
- 查询超时
- 数据损坏

**存储故障**：
- R2 不可用
- KV 同步失败
- 存储配额已满

### 4.2 故障排查流程

```
1. 确认问题（监测告警、用户反馈）
2. 评估影响范围（多少用户？什么功能？）
3. 启用 P0 响应（通知团队）
4. 收集信息（日志、指标、错误堆栈）
5. 分析根本原因（RCA）
6. 快速修复或降级
7. 验证修复有效
8. 通知用户
9. 事后总结（Postmortem）
```

### 4.3 常见故障及解决方案

**故障**：WebSocket 连接断开
```
症状：用户无法接收实时消息
排查：
1. 检查网络连接：wrangler tail --env production
2. 查看 Durable Objects 日志
3. 检查会话是否过期
4. 检查 Worker 内存使用

解决：
1. 重启 Durable Objects（清除内存）
2. 清理过期会话
3. 增加 Worker 内存（如支持）
4. 降级到轮询模式（临时方案）
```

**故障**：消息无法存储
```
症状：消息发送成功但未保存到数据库
排查：
1. 检查 D1 连接状态
2. 查看数据库磁盘空间
3. 检查数据库锁定

解决：
1. 重新连接 D1
2. 清理旧消息（如配置硬删除）
3. 杀死长期锁定的事务
```

**故障**：文件上传失败
```
症状：上传返回 500 错误
排查：
1. 检查 R2 连接
2. 检查存储配额
3. 查看 Worker 日志

解决：
1. 增加 R2 存储配额
2. 清理过期文件
3. 检查文件权限
```

### 4.4 快速降级方案

若关键功能不可用，可采取降级策略：

| 故障 | 降级方案 |
|------|---------|
| WebSocket 不可用 | 切换到 HTTP 轮询（每 1 秒）|
| 消息发送失败 | 本地缓存，待恢复后重试 |
| 文件上传故障 | 禁用上传功能，返回临时错误 |
| 登录失败 | 使用缓存的会话（如有） |

## 5. 备份与恢复

### 5.1 备份策略

**D1 数据库**：
- 自动备份：Cloudflare 每日
- 手动备份：每周导出（推荐）

```bash
# 导出数据库
wrangler d1 execute edgechat-db \
  "SELECT * FROM messages" > backup-messages.sql

# 导出所有表
wrangler d1 execute edgechat-db \
  ".dump" > backup-full.sql
```

**KV 会话**：
- 无备份需求（会话可丢失，用户重新登录）
- 无需持久化

**R2 文件**：
- S3 兼容备份
- 定期上传到异地存储

### 5.2 恢复流程

**数据库恢复**：
```bash
# 1. 创建新的数据库
wrangler d1 create edgechat-db-recovery

# 2. 从备份恢复数据
wrangler d1 execute edgechat-db-recovery --file backup-full.sql

# 3. 验证数据完整性
wrangler d1 execute edgechat-db-recovery "SELECT COUNT(*) FROM messages"

# 4. 更新 wrangler.toml，切换到新数据库

# 5. 重新部署 Worker
wrangler deploy --env production
```

**RTO 和 RPO**：
- RTO（恢复时间目标）：< 1 小时
- RPO（恢复点目标）：< 1 天

## 6. 性能优化

### 6.1 数据库优化

**分析慢查询**：
```sql
-- 查询耗时统计
SELECT query, COUNT(*) as count, AVG(duration_ms) as avg_duration
FROM query_log
WHERE duration_ms > 100
GROUP BY query
ORDER BY avg_duration DESC;
```

**优化建议**：
- 对常用查询字段添加索引
- 避免 JOIN 大表
- 使用分页而非一次加载全部
- 定期分析表统计信息

### 6.2 缓存优化

**缓存层**：
```
热数据（频繁访问） -> KV 缓存（< 1 秒）
温数据（偶尔访问）   -> D1 查询（< 100ms）
冷数据（很少访问）   -> 归档（外部存储）
```

**缓存淘汰**：
- TTL：默认 1 小时
- LRU：超出容量时删除最旧数据

### 6.3 Worker 优化

**减少计算**：
- 避免 N+1 查询
- 批量操作
- 使用异步非阻塞 I/O

**减少内存**：
- 定期清理 Durable Objects
- 不在内存中存储完整消息历史
- 及时释放连接

## 8. 容量规划

### 8.1 存储容量

**D1 数据库容量评估**：
- 每条消息：约 500 字节
- 消息总量估算：日活 \* 消息/天 \* 保留天数
- 示例：1000 日活 \* 100 消息 \* 7 天 = 700,000 条消息 = 350MB

**R2 文件存储**：
- 平均文件大小：2MB（混合视频和图片）
- 存储容量：日活 \* 日上传数 \* 平均大小

### 8.2 性能容量

**同时连接数**：
- 每个 Durable Object 支持 1000+ 并发连接
- 频道数量 = 同时连接数 / 平均成员数

**请求速率**：
- Workers 自动扩展，无固定上限
- CDN 可支持 Gbps 级别的流量

### 8.3 升级计划

根据实际使用情况，定期评估是否需要升级：
- 存储容量增加
- 数据库副本增加（如支持）
- 区域扩展

## 9. 灾难恢复

### 9.1 灾难恢复计划（DRP）

**灾难场景**：
1. 数据中心故障（Cloudflare 处理，无需干预）
2. 数据库损坏
3. 应用代码错误导致数据修改
4. 安全漏洞导致数据泄露

**恢复过程**：
```
1. 启用 DR 模式（不接受新请求）
2. 从备份恢复数据库
3. 验证数据完整性
4. 部署正确的代码版本
5. 逐步恢复流量
6. 监测和验证
```

### 9.2 恢复时间目标（RTO）

- 数据库故障：< 1 小时
- 应用故障：< 30 分钟（重新部署）
- 数据损坏：< 4 小时

### 9.3 恢复点目标（RPO）

- 无法接受的损失：< 24 小时（每日备份）
- 推荐的 RPO：< 1 小时（每小时备份）

### 10.2 常见问题（FAQ）

**Q：系统如何处理高并发？**
A：Workers 自动扩展，Durable Objects 可支持 1000+ 并发连接，D1 支持高并发查询。

**Q：如何确保消息不丢失？**
A：消息立即写入 D1（同步），所有写操作使用事务，定期备份。

**Q：如何回滚错误的部署？**
A：使用 `wrangler rollback`，或重新部署前一个版本。

**Q：存储空间不足时怎么办？**
A：删除过期消息（软删除后 7 天硬删除），清理过期文件，升级 R2 配额。
